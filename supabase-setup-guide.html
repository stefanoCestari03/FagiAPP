1 Crea il Progetto Supabase

1
Vai su supabase.com e accedi

Crea un account gratuito se non ce l'hai. Il piano Free supporta fino a 500 MB di database e 50.000 richieste/mese — più che sufficiente per questa app.

2
Crea un nuovo progetto

Clicca New Project, scegli un'organizzazione, inserisci:

  • Name: fagitana-cantieri
  • Database Password: crea una password forte e salvala
  • Region: West EU (Ireland) — più vicina all'Italia
3
Attendi la creazione (circa 1-2 minuti)

Supabase provisionerà il database PostgreSQL. Una volta pronto, vai in Project Settings → API e copia: Project URL e anon public key. Ti serviranno al passo 6.

2 Crea le Tabelle del Database

Vai su SQL Editor nel menu laterale di Supabase ed esegui questo script completo. Crea tutte le tabelle, i constraint, gli indici e i trigger necessari.

⚠️
Esegui in SQL Editor, non in Table Editor

Copia l'intero blocco SQL, incollalo nel SQL Editor e premi Run (Ctrl+Enter). Le istruzioni vanno eseguite in ordine.

SQL — Schema Completo 
-- ============================================================
--  FagiAPP Cantieri — Schema Database Supabase
--  Eseguire intero blocco nel SQL Editor
-- ============================================================

-- Abilita l'estensione UUID (già presente su Supabase)
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- ============================================================
--  TABELLA: operai
--  Lista dei 7 lavoratori fissi dell'azienda
-- ============================================================
CREATE TABLE IF NOT EXISTS operai (
  id          uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  nome        text        NOT NULL,
  cognome     text        NOT NULL,
  attivo      boolean     NOT NULL DEFAULT true,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- ============================================================
--  TABELLA: cantieri
--  Cantieri edili dell'azienda
-- ============================================================
CREATE TABLE IF NOT EXISTS cantieri (
  id          uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  nome        text        NOT NULL,
  comune      text,
  indirizzo   text,
  fase        text        CHECK (fase IN (
                'Muratura', 'Carpenteria', 'Rifinitura',
                'Impianti', 'Copertura', 'Fondamenta',
                'Intonaco', 'Pavimentazione'
              )),
  stato       text        NOT NULL DEFAULT 'attivo'
                CHECK (stato IN ('attivo', 'completato', 'sospeso')),
  note        text,
  created_by  uuid        REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at  timestamptz NOT NULL DEFAULT now()
);

-- Indice per filtrare cantieri per stato (attivo/completato)
CREATE INDEX idx_cantieri_stato ON cantieri(stato);

-- ============================================================
--  TABELLA: giornate
--  Registrazione giornaliera per ogni cantiere
-- ============================================================
CREATE TABLE IF NOT EXISTS giornate (
  id                uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  cantiere_id       uuid        NOT NULL REFERENCES cantieri(id) ON DELETE CASCADE,
  data              date        NOT NULL,
  meteo             text,
  ora_inizio        time        DEFAULT '07:00',
  ora_fine          time        DEFAULT '17:00',
  ora_pausa_inizio  time        DEFAULT '12:00',
  ora_pausa_fine    time        DEFAULT '13:00',
  fase              text        CHECK (fase IN (
                      'Muratura', 'Carpenteria', 'Rifinitura',
                      'Impianti', 'Copertura', 'Fondamenta',
                      'Intonaco', 'Pavimentazione'
                    )),
  avanzamento       integer     DEFAULT 0
                      CHECK (avanzamento BETWEEN 0 AND 100),
  note_attivita     text,
  note_problemi     text,
  note_prossimi     text,
  created_by        uuid        REFERENCES auth.users(id) ON DELETE SET NULL,
  created_at        timestamptz NOT NULL DEFAULT now(),
  -- Una sola registrazione per cantiere per giorno
  UNIQUE (cantiere_id, data)
);

-- Indici per le query più frequenti
CREATE INDEX idx_giornate_cantiere   ON giornate(cantiere_id);
CREATE INDEX idx_giornate_data       ON giornate(data DESC);
CREATE INDEX idx_giornate_data_range ON giornate(data, cantiere_id);

-- ============================================================
--  TABELLA: presenze
--  Presenze operai per ogni giornata
-- ============================================================
CREATE TABLE IF NOT EXISTS presenze (
  id           uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  giornata_id  uuid        NOT NULL REFERENCES giornate(id) ON DELETE CASCADE,
  operaio_id   uuid        NOT NULL REFERENCES operai(id) ON DELETE RESTRICT,
  stato        text        NOT NULL DEFAULT 'presente'
                 CHECK (stato IN ('presente', 'parziale', 'assente')),
  ora_entrata  time,
  ora_uscita   time,
  note         text,
  -- Un operaio una sola presenza per giornata
  UNIQUE (giornata_id, operaio_id)
);

CREATE INDEX idx_presenze_giornata ON presenze(giornata_id);
CREATE INDEX idx_presenze_operaio  ON presenze(operaio_id);

-- ============================================================
--  TABELLA: materiali
--  Materiali utilizzati per ogni giornata
-- ============================================================
CREATE TABLE IF NOT EXISTS materiali (
  id           uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  giornata_id  uuid        NOT NULL REFERENCES giornate(id) ON DELETE CASCADE,
  nome         text        NOT NULL,
  quantita     text,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_materiali_giornata ON materiali(giornata_id);

-- ============================================================
--  TABELLA: mezzi
--  Mezzi e attrezzature per ogni giornata
-- ============================================================
CREATE TABLE IF NOT EXISTS mezzi (
  id           uuid        PRIMARY KEY DEFAULT uuid_generate_v4(),
  giornata_id  uuid        NOT NULL REFERENCES giornate(id) ON DELETE CASCADE,
  nome         text        NOT NULL,
  utilizzo     text,
  created_at   timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX idx_mezzi_giornata ON mezzi(giornata_id);

-- ============================================================
--  TRIGGER: aggiorna updated_at automaticamente su giornate
-- ============================================================
CREATE OR REPLACE FUNCTION handle_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Aggiungi colonna updated_at e trigger su giornate
ALTER TABLE giornate ADD COLUMN IF NOT EXISTS updated_at timestamptz;

CREATE OR REPLACE TRIGGER giornate_updated_at
  BEFORE UPDATE ON giornate
  FOR EACH ROW EXECUTE FUNCTION handle_updated_at();

3 Configura la Sicurezza (RLS)

Row Level Security garantisce che solo gli utenti autenticati possano leggere e scrivere i dati. Esegui questo script dopo quello dello schema.

🔒
Importante: senza RLS i dati sono pubblici

Supabase di default espone le tabelle pubblicamente via API. L'RLS è obbligatorio per proteggere i dati dell'azienda.

SQL — Row Level Security 
-- ============================================================
--  RLS POLICIES — Solo utenti autenticati hanno accesso
-- ============================================================

-- Abilita RLS su tutte le tabelle
ALTER TABLE operai     ENABLE ROW LEVEL SECURITY;
ALTER TABLE cantieri   ENABLE ROW LEVEL SECURITY;
ALTER TABLE giornate   ENABLE ROW LEVEL SECURITY;
ALTER TABLE presenze   ENABLE ROW LEVEL SECURITY;
ALTER TABLE materiali  ENABLE ROW LEVEL SECURITY;
ALTER TABLE mezzi      ENABLE ROW LEVEL SECURITY;

-- ------------------------------------------------
--  OPERAI: solo lettura per utenti autenticati
--  (gli operai non cambiano — li gestisce solo un admin)
-- ------------------------------------------------
CREATE POLICY "operai_select" ON operai
  FOR SELECT TO authenticated USING (true);

-- ------------------------------------------------
--  CANTIERI: lettura e scrittura per autenticati
-- ------------------------------------------------
CREATE POLICY "cantieri_select" ON cantieri
  FOR SELECT TO authenticated USING (true);

CREATE POLICY "cantieri_insert" ON cantieri
  FOR INSERT TO authenticated
  WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "cantieri_update" ON cantieri
  FOR UPDATE TO authenticated USING (true);

-- ------------------------------------------------
--  GIORNATE: lettura e scrittura per autenticati
-- ------------------------------------------------
CREATE POLICY "giornate_select" ON giornate
  FOR SELECT TO authenticated USING (true);

CREATE POLICY "giornate_insert" ON giornate
  FOR INSERT TO authenticated
  WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "giornate_update" ON giornate
  FOR UPDATE TO authenticated USING (true);

CREATE POLICY "giornate_delete" ON giornate
  FOR DELETE TO authenticated USING (true);

-- ------------------------------------------------
--  PRESENZE, MATERIALI, MEZZI: lettura e scrittura
-- ------------------------------------------------
CREATE POLICY "presenze_all" ON presenze
  FOR ALL TO authenticated USING (true) WITH CHECK (true);

CREATE POLICY "materiali_all" ON materiali
  FOR ALL TO authenticated USING (true) WITH CHECK (true);

CREATE POLICY "mezzi_all" ON mezzi
  FOR ALL TO authenticated USING (true) WITH CHECK (true);
✅
Strategia scelta: tutti gli autenticati vedono tutto

L'app è ad uso interno dell'azienda. Ogni utente con account può vedere e modificare tutti i cantieri — non serve un isolamento per-utente. Se in futuro vuoi che ogni utente veda solo le proprie giornate, sostituisci USING (true) con USING (created_by = auth.uid()) nella policy di giornate.

5 Configura l'Autenticazione

L'app usa Email + Password via Supabase Auth. Devi creare gli utenti manualmente dalla dashboard (non c'è form di registrazione pubblica — corretto per un'app aziendale interna).

1
Vai su Authentication → Users nella dashboard Supabase

Clicca il pulsante "Invite user" oppure "Add user".

2
Crea l'utente amministratore

Inserisci email e password per ogni persona che dovrà accedere all'app. Esempio: caposquadra@fagitana.it

3
Configura le impostazioni email (opzionale)

In Authentication → Email Templates puoi personalizzare le email di conferma. Per un'app interna, puoi anche disabilitare la conferma email in Authentication → Providers → Email → Confirm email.

4
Imposta il Site URL

In Authentication → URL Configuration, imposta il Site URL all'URL dove è deployata la tua app (es. https://fagitana.netlify.app oppure http://localhost:5173 in sviluppo).

⚠️
Non serve "Disable email confirmation" per il primo test

Se crei l'utente tramite Add user → Create new user (non Invite), Supabase lo crea già confermato. Puoi usare subito email e password per il login nell'app.

6 Collega l'App al Database Reale

Ora devi sostituire il mock localStorage con il vero client Supabase. Ci sono due file da modificare.

6.1 — Crea il file .env con le credenziali

Il progetto ha già un .env.example. Crea il file .env nella cartella fagitana-cantieri-app/fagitana-app/:

.env — Variabili d'ambiente 
# Copia questi valori da: Supabase Dashboard → Project Settings → API
VITE_SUPABASE_URL=https://xxxxxxxxxxx.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
🔐
Non committare mai il file .env su Git

Verifica che .env sia in .gitignore. Usa solo la anon key (pubblica), mai la service_role key (admin) nel frontend.

6.2 — Sostituisci il client Supabase mock

Apri il file src/lib/supabase.js e sostituisci tutto il contenuto con questo:

src/lib/supabase.js — Client Reale 
import { createClient } from '@supabase/supabase-js'

const supabaseUrl  = import.meta.env.VITE_SUPABASE_URL
const supabaseKey  = import.meta.env.VITE_SUPABASE_ANON_KEY

export const supabase = createClient(supabaseUrl, supabaseKey)

6.3 — Installa il pacchetto Supabase

Apri il terminale nella cartella fagitana-cantieri-app/fagitana-app/ e installa il client:

Terminale 
npm install @supabase/supabase-js

✅

Il mock è stato progettato per imitare esattamente l'API Supabase. Cambiando solo src/lib/supabase.js, tutte le query in Dashboard.jsx, Calendario.jsx, Cantieri.jsx e NuovaGiornata.jsx funzioneranno senza modifiche.

7 Verifica e Collaudo

Usa queste query SQL per verificare che tutto funzioni correttamente dal SQL Editor di Supabase.

SQL — Query di verifica 
-- 1. Verifica operai inseriti
SELECT id, nome, cognome, attivo FROM operai ORDER BY cognome;

-- 2. Verifica cantieri
SELECT id, nome, comune, stato, created_at FROM cantieri;

-- 3. Conta giornate per cantiere
SELECT
  c.nome AS cantiere,
  COUNT(g.id) AS giornate_registrate,
  MAX(g.data) AS ultima_giornata
FROM cantieri c
LEFT JOIN giornate g ON g.cantiere_id = c.id
GROUP BY c.id, c.nome
ORDER BY c.nome;

-- 4. Dettaglio ultima giornata con presenze
SELECT
  g.data,
  c.nome AS cantiere,
  g.meteo,
  g.fase,
  g.avanzamento || '%' AS avanzamento,
  o.nome || ' ' || o.cognome AS operaio,
  p.stato
FROM giornate g
JOIN cantieri c ON c.id = g.cantiere_id
LEFT JOIN presenze p ON p.giornata_id = g.id
LEFT JOIN operai o ON o.id = p.operaio_id
ORDER BY g.data DESC, o.cognome
LIMIT 20;

-- 5. Verifica RLS attivo su tutte le tabelle
SELECT
  schemaname,
  tablename,
  rowsecurity AS rls_enabled
FROM pg_tables
WHERE schemaname = 'public'
  AND tablename IN ('operai','cantieri','giornate','presenze','materiali','mezzi');
